1 — Context: How These Packages Enter the Image
The KServe operator binary is compiled with CGO_ENABLED=0 (a fully static Go binary) and packaged using gcr.io/distroless/static:nonroot as the runtime base image. The operator itself contains no Debian packaging; however, the distroless image is constructed by Google from a curated subset of Debian packages. netbase and base-files are pulled in as transitive dependencies of the distroless build chain and materialise as read-only configuration files inside the container filesystem.

Key point: No source code from either package is incorporated into the operator binary. No executable from either package runs inside the container. They exist solely as filesystem artifacts (e.g. /etc/services, /etc/protocols, /etc/debian_version).

🔍 Review Board Assessment — netbase
GPL-2.0-only is a strong copyleft license; however, copyleft obligations are triggered by distribution of derived works — i.e., modifying and distributing the GPL-licensed source code itself. Shipping an unmodified Debian package as a data file inside a container image is equivalent to distributing a pre-built Debian system: the operator is not a derivative work of netbase. No source-code disclosure of the operator is required. The only obligation is attribution (referencing upstream Debian as the source), which the distroless project fulfils by embedding the /usr/share/doc/netbase/copyright file inside every image layer.

🔍 Review Board Assessment — base-files
GPL-2.0-or-later with the same reasoning as netbase. The -or-later clause is more permissive than -only as it allows future GPL versions. base-files is a standard infrastructure package shipped with every Debian-based container image worldwide — its presence is universal and its compliance posture is well-established. No operator source disclosure is required. Attribution is embedded in the image via the standard Debian copyright mechanism.

5 — Compliance Obligations
  • ✓
    Acknowledge presence in SBOM / release notes. Both packages must be listed in any Software Bill of Materials (SBOM) generated for each release. The distroless project's own SBOM tooling (compatible with SPDX and CycloneDX) can be used.
  • ✓
    Point to upstream source on request. If a downstream customer or partner asks for source code, direct them to https://packages.debian.org/. You are not required to host copies yourself.
  • ✗
    You do NOT need to open-source your operator. GPL copyleft does not extend to applications that merely run in a container alongside GPL-licensed config files.
  • ✗
    You do NOT need to ship the GPL license text yourself. The distroless image already embeds /usr/share/common-licenses/GPL-2 and per-package copyright files inside the image layers, satisfying the attribution requirement.
💡
6 — Recommendations for Future Releases
🔒 Pin Image by Digest

Replace :nonroot tag with a pinned SHA-256 digest in the Dockerfile to prevent silent base-image updates that could introduce new packages between releases.

📄 Generate SBOM per Release

Run syft or docker sbom on the final operator image as part of the CI pipeline and attach the SPDX/CycloneDX file as a release artifact.

🔍 Use Trivy / Grype for CVE Scanning

Run vulnerability scanning on the image after each build to detect any CVEs in the distroless layer packages, including netbase and base-files.

📬 Establish a Source-Request Process

Document an internal process for responding to GPL source-code requests from customers, pointing to packages.debian.org as the canonical upstream.
